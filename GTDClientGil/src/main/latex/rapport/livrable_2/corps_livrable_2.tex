	\chapter{Spécification d'exigences logicielles}
	\minitoc

		\section{Introduction}

			\subsection{Objectif}

Ce deuxième livrable se focalise sur la spécification des exigences logicielles de notre application. Nous sommes concernés par l'application cliente de ce projet centrée autour de la méthode GTD. Nous considérons tout au long de ce document que les conditions sont idéales pour la partie que nous allons décrire. Le terme idéal signifie ici que pour fonctionner, notre client a besoin d'un ensemble de données et de messages associés. Ces messages arrivent toujours à destination, les données produites sont valides, et le serveur distant est toujours disponible.

		%\section{Conventions}

			\subsection{Audience}
Ce document s'adresse à un chef de projet (ou à un professeur).

			\subsection{Portée du document}
Ce document se focalise sur l'application dite cliente d'un système.

			\subsection{Définitions, acronymes et abréviations}
\begin{itemize}
	\item GTD : Getting Things Done (méthode de gestion de tâches quotidienne ou professionelle dans un système distribué)
	\item Client GTD : application locale généralement stockée sur le poste de travail d'un utilisateur.
	\item IHM GTD : C'est l'Interface Homme-Machine permettant à l'utilisateur d'utiliser l'application.
\end{itemize}

			\subsection{Organisation du chapitre}

Dans ce deuxième livrable, nous avons exprimé tous les besoins de notre système GTD (besoins fonctionnels et besoins non fonctionnels). Dans un premier temps, nous avons détaillé les cas d'utilisation du premier livrable en détaillant des sous-cas. Dans un second temps, nous avons exprimé, de façon littérale, tous les aspects dont notre application a besoin. Cette étape est nécessaire dans l'analyse et la spécification de notre système puisqu'elle permet de se rendre compte de notre vision en tant que client GTD du système.

		\subsection{Fonctions du produit}
Le client GTD doit permettre à un utilisateur de gérer efficacement des tâches. Il doit pouvoir les retrouver très simplement, en effectuant divers tris, ou en les organisant en projets.

		\subsection{Caractéristiques et classes d'utilisateurs}
Cette application est destinée au grand public. N'importe qui doit pouvoir créer ses propres tâches et cela très rapidement. Cependant, des notions minimales sont requises pour bien exploiter les différentes fonctionalités du logiciel. Ainsi, l'utilisateur lambda doit être familié avec les concepts de contexte et de tri.

		\subsection{Environnement opérationnel}
%<Describe the environment in which the software will operate, including the hardware platform, operating system and versions, and any other software components or applications with which it must peacefully coexist.>
	\paragraph{}
Le logiciel doit pouvoir s'éxécuter indépendamment du matériel sur lequel l'utilisateur travail. En effet, la configuration minimale demande une carte graphique ayant 65535 couleurs, un processeur 1GHz et un espace disque de 5Go. Cette application se destine à une poste traditionnel type Desktop. Une utilisation sur ordinateur portable est possible aussi. Pour cela, l'IHM doit être conçue de façon à être utilisée sur des résolutions peu élevées.

	\paragraph{}
Le client GTD se base sur le langage Java. Une machine virtuelle doit donc être disponible sur le poste cible.

		\subsection{Contraintes de conception et d'implémentation}
L'implémentation ne pourra se faire que lorsque tous les groupes de ce projet multi-module seront d'accord sur les interfaces à établir. Comme cela n'arrivera probablement pas, le groupe gérant la conception du serveur GTD imposera les spécifications qui lui sont propre.

		\subsection{Documentation utilisateur}
Le GTD aura une documentation interne permettant à l'utilisateur de retrouver rapidement comment réaliser ses actions. Cette documentation reprend les différents cas d'utilisation décrits prochainement. Cette documentation peut être soit au format papier, soit au format électronique. Une documentation papier permet aux utilisateurs ayant peu d'expérience, avec un environnement multi-fenêtré, d'avoir sous les yeux toutes les informations utiles.

 		\subsection{Synchroniser les tâches}
En déplacement ou depuis un autre poste, que ce soit un client GTD ou bien un client web, l'utilisateur doit pouvoir retrouver l'ensemble de ses tâches. La fonctionnalité de synchronisation offre la possibilité de synchroniser le contenu local avec un serveur GTD distant. N'étant pas un gestionnaire de version, cette fonctionnalité ne pourra s'appuyer que sur les dates de dernière modification des tâches.\\
			\subsubsection{Description}

\begin{usecase}{Synchroniser les tâches}

\begin{information}
\item[Goal in the context:]
L'utilisateur souhaite synchroniser son client GTD local avec le serveur GTD.

\item[Scope:]
Client et serveur GTD

\item[Level:]
Sous-fonction

\item[Success End Condition:]
Les tâches de l'utilisateur sont identiques sur le serveur et le client local. Le nombre de tâche est aussi identique.

\item[Failed End Condition:]
La synchronisation échoue : le client ne parvient pas à obtenir les dernières modifications disponibles sur le serveur, ou bien il ne parvient pas à envoyer les mises à jour (nouvelles tâches, changement d'état, de date, etc.).

\item[Primary actor:]
L'utilisateur

\item[Trigger:]
Le client GTD effectue la synchronisation à chacun de ses démarrages et fermetures.

\item[Precondition:]

\item[Postcondition:]
	
\end{information}

\begin{scenario}
\item L'utilisateur lance le client GTD.
\item Le client GTD se connecte au serveur.
\item Le client GTD demande au serveur sa liste de tâches.
\item Le client GTD récupère les tâches plus récentes sur le serveur.
\item Le client GTD envoi les tâches plus récentes en local au serveur.
\item Le client GTD achève la synchronisation et se déconnecte.\\
\end{scenario}

\begin{extension}
\item[2a] Le client GTD ne parvient pas à créer la liaison avec le serveur
	\item[2a1] Le client GTD interrompt la synchronisation
\end{extension}

\begin{subvariation}
	\item Fermeture du client GTD 
\end{subvariation}

	\section{Autres exigences non-fonctionnelles}

		\subsection{Utilisabilité}

			\subsubsection{Facteurs Humains}

Quelque soit le type d'utilisateur (novice ou professionnel en informatique) qui utilise notre logiciel, il ne doit pas avoir de problème d'utilisation. Le logiciel doit être simple. On ne doit pas demander d'actions complexes à l'utilisateur. Si l'utilisateur n'est pas au point sur la création de tâches et de projets directement, il pourra changer de méthode de travail. En effet, nous prévoyons une sorte d'agenda (probablement dans un onglet) contenant tous les jours d'un mois donné afin de ranger les taches et les projets plus facilement. 

			\subsubsection{Esthétique}

Lorsque l'utilisateur utilise notre logiciel, il doit être agréablement surpris. L'interface graphique doit sortir de l'ordinaire tout en restant cohérente.

			\subsubsection{Cohérence de l'IHM}

L'utilisateur doit être à l'aise avec l'interface graphique. En effet, il doit trouver rapidement et simplement les actions qu'il doit effectuer. L'utilisateur doit avoir le sentiment de travailler dans des conditions qu'il connait déjà. En effet, les automatismes des IHM doivent être respectés (la disposition des différents éléments graphique, couleurs,...). L'utilisation de couleurs communes est privilégiée car l'utilisateur ne doit pas être fatigué à cause de l'IHM.

			\subsubsection{Aide en ligne et contextuelle}

À tout moment, l'utilisateur, si besoin est, doit savoir ce qu'il fait. C'est pour cela qu'un commentaire sera affiché si l'utilisateur hésite à faire une action.

			\subsubsection{Assistants et agents}

Nous n'utiliserons pas d'agents puisque cela peut devenir penible au démarrage du GTD. En effet, nous estimons que la plupart des utilisateur desactiverons cet agent donc nous ne l'implementerons pas. Par contre, nous pouvons créer une fenêtre d'assistance dont le contenu guidera pas à pas l'utilisateur, s'il souhaite être assisté (en cochant par exemple sur une case assistance).

			\subsubsection{Supports de formation}

Afin d'aiguiller l'utilisateur dans ses choix, des didacticiels doivent être fournis. Un format vidéo est appréciable. Si besoin, nous pouvons aussi activer le didactitiel lors du démarrage de l'application et l'utilisateur decidera de le passer ou non.


			\subsubsection{Autres}

Dans un cadre plus professionnel, le temps de formation nécessaire à un utilisateur normal ou expert avant d'être productif sera le même. Une première fois l'application testée, l'amateur ou l'expert en informatique pourra ré-utiliser parfaitement l'application.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

		\subsection{Fiabilité}

			\subsubsection{Fréquence et gravité des échecs}

Nous autoriserons au plus un incident toute les heures, afin de ne pas exaspérer l'utilisateur.

			\subsubsection{Récupérabilité}

Dans n'importe quelles circonstances, l'utilisateur pourra revenir d'un cran en arrière si quelque chose de non voulu se produit.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

		\subsection{Exigences de performance}

			\subsubsection{Rapidité}

Quelque soit les actions faites par l'utilisateur, chaque action doit prendre moins de 30 secondes (que se soit la connexion à son compte, la création d'une tâche, \dots). Même le temps d'attente doit être inférieur à 30 secondes afin que l'utilisateur ne se sente pas délaissé et ennuyé (lors par exemple de la synchronisation avec le serveur, l'affichage par tri, \dots).

			\subsubsection{Efficacité}

Bien évidemment, toutes les fonctionnalités dites "primaires" doivent être respectées  et implémentées (créer une tâche,...). Il se peut que l'on puisse rajouter quelques actions (choix de couleur de l'interface graphique, tutoriel interactif, \dots).

			\subsubsection{Disponibilité}

Le logiciel doit pouvoir s'utiliser 24h/24 et 7j/7. Pour cela, le serveur doit être disponible dans ces mêmes horaires sauf période de maintenance (pour cela le serveur préviendra avec un message d'une maintenance durant le temps requis).

			\subsubsection{Temps de réponse}

lié à la rapidité, voir Performance-Rapidité)

			\subsection{Temps de récupération d'une erreur}

Le temps de récupération d'une erreur doit être instanné alors chaque cas d'erreur devra être traité.

			\subsubsection{Consommation des ressources}

La consommation sera la plus basse possible. En effet, il ne s'agit pas d'un logiciel en 3 dimensions capable d'effectuer des super calculs, donc le GTD devra consommer autant qu'un logiciel de gestion.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

		\subsection{Maintenabilité}

Lors de l'implémentation, il sera aisé d'ajouter des fonctionnalités, d'en supprimer ou d'en éditer grâce à l'utilisation d'interface.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

		\subsection{Exigences de sécurité}

On peut prévoir une réplication des données toutes les nuits et des sauvegardes sur différents serveurs. On sauvegardera les actions de l'utilisateur à intervalles réguliers. Par contre, les données envoyées au serveur ne seront pas cryptées car nous considérons l'application GTD comme non confidentielle.

	\section{Autres exigences}

		\subsection{Adaptabilité}

			\subsubsection{Testabilité}

L'implémentation logiciel aura en parallèle une implémentation de test afin de tester la totalité des fonctionnalités du logiciel, mais aussi la gestion du code, \dots

			\subsubsection{Extensibilité}

D'autres logiciels pourront utiliser notre logiciel ou l'exécuter au sein même d'un autre logiciel, nous pensons donc le créer sous forme d'un plugin ou pouvoir le mettre sous forme d'un plugin si ce n'est pas faisable immédiatemment.

			\subsubsection{Comptabilité}

Le logiciel sera indépendant de toutes plateforme et de tout autres services externes. Ainsi, il pourra être associé à d'autres application GTD si l'utilisateur en utilise plusieurs par exemple.

			\subsubsection{Configurabilité}

L'utilisateur pourra, selon son désir, remanié l'interface graphique (la couleur, les barres de menu,...) et aussi la langue.

			\subsubsection{Installabilité}

Le logiciel fonctionnant sur n'importe quel machine, nous proposons de crée une archive (par système d'exploitation) avec le logiciel dans son intégralité et son README correspondant(c'est à a dire pas d'installation complexe). Ce système permet aux utilisateurs amateurs de pouvoir installer facilement l'application sans que le système d'exploitation autorise ou non cette application.

			\subsubsection{Internationalisation}

Nous implémenterons dans le logiciel la capacité de choisir sa langue en cliquant respectivement sur des "petits drapeaux" en haut à gauche de la fenêtre (ne pouvant effectuer toutes les traductions de langues au monde, nous nous restreindrons aux langues Anglais, Français, Espagnol.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

		\subsection{Déploiement}

Le logiciel doit pouvoir s'éxécuter sur n'importe quel plateforme (Windows/Linux/Mac) et pour cela, nous allons utiliser le langage Java puisqu'il est adapté à toutes formes de plateforme. De plus, ce langage est entré dans les normes. L'utilisateur doit avoir la machine virtuelle Java d'installé sur son poste pour pouvoir utiliser ce logiciel. S'il ne la possède pas, une version récente et stable doit être fournie avec le support d'installation, ou facilement téléchargeable (voire intégré dans une archive) si le logiciel est totalement dématérialisé.
